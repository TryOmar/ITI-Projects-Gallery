ITI Projects Gallery

Technical Requirements Document
Version 1.0
Document Date: December 2024

1. Project Overview

ITI Projects Gallery is a web-based gallery system designed to showcase projects created by ITI students during their 9-month training program. The system is architected to provide maximum ease of use for project submission and editing while maintaining a lightweight infrastructure without traditional backend requirements.

The system leverages HTML, CSS, and JavaScript for the frontend interface, with Google Sheets and Google Apps Script serving as the backend infrastructure and API layer respectively.

2. Project Objectives

  1. Enable ITI students to submit their projects through a simple web interface
  2. Provide students with the ability to edit their own project submissions post-submission
  3. Display all approved projects in a publicly accessible gallery interface
  4. Maintain a lightweight backend infrastructure using only Google Sheets
  5. Implement admin moderation capabilities without complex role-based access control systems

3. Technology Stack

3.1 Frontend Technologies

  • HTML5
  • CSS3
  • JavaScript (Vanilla JavaScript or lightweight framework)

3.2 Backend Infrastructure

  • Google Sheets (serves as database layer)
  • Google Apps Script (serves as API layer)

3.3 Hosting Platform

  • Static hosting service (GitHub Pages, Netlify, or equivalent)

4. System Architecture

4.1 Application Pages

The system consists of five HTML pages, three required and two optional:

Page Name Filename Status Primary Function
Gallery Page index.html Required Display all visible projects
Submission Page submit.html Required Submit new projects
Edit Page edit.html Required Edit existing projects
Admin Page admin.html Optional Moderate project visibility
About Page about.html Optional System documentation

5. Page Specifications

5.1 Gallery Page (index.html)

Purpose

Display all projects marked as visible in the public gallery interface.

Functional Requirements

  • Fetch project data from Google Sheets via Google Apps Script API
  • Display only projects where the Visible field is set to TRUE
  • Render project information in card format
  • Implement filtering mechanisms by Status, Framework, and Team Member
  • Provide search functionality across project data

Project Card Display Elements

  • Project Title
  • Team Members (displayed as GitHub handles)
  • Project Description
  • Status Badge (Not Started, In Progress, Completed)
  • Frameworks and Technology Stack
  • Links (Demo, GitHub Repository, Application URL)
  • Screenshot (optional)

User Interface Components

  • Filter controls for Status, Framework, and Team Member
  • Search bar for text-based queries
  • Responsive card grid layout

5.2 Project Submission Page (submit.html)

Purpose

Provide an interface for students to submit new project entries to the system.

Form Fields Specification

Field Name Type Required Description
Project Title Text Input Yes Name of the project
Team Members Text Input No GitHub handles of team members
Description Textarea No Project description
Demo/GitHub/App Link URL Input No Project-related URLs
Frameworks/Tech Stack Text Input No Technologies used
Status Dropdown Yes Not Started, In Progress, Completed
Screenshot File/URL Input No Project screenshot
Email Email Input No Contact email for editing rights

Submission Process

  1. User completes and submits the form
  2. JavaScript validates form data client-side
  3. Data is transmitted to Google Apps Script via fetch/AJAX
  4. Apps Script appends new row to Google Sheet
  5. Success or error message displayed to user

5.3 Project Edit Page (edit.html)

Purpose

Allow students to modify their existing project submissions.

Project Identification

Projects can be identified using one of the following methods:

  • Email address + Project Title (alternative method)
  • Unique Project ID (recommended method)

Functional Requirements

  • Provide input fields for project identification
  • Fetch existing project data from Google Sheet via Apps Script
  • Pre-populate form fields with current project data
  • Submit updated data via Apps Script
  • Update existing row in Google Sheet (not append new row)
  • Display confirmation message upon successful update

Edit Process Flow

  1. User enters project identification information
  2. System retrieves project data from Google Sheet
  3. Form fields are pre-populated with existing data
  4. User modifies desired fields
  5. User submits updated form
  6. System updates corresponding row in Google Sheet
  7. Confirmation displayed to user

5.4 Admin Page (admin.html) - Optional

Purpose

Provide administrative interface for project moderation and visibility control.

Functional Requirements

  • Display complete list of all projects (including hidden projects)
  • Toggle project visibility (TRUE/FALSE in Visible field)
  • Delete projects (optional feature)
  • Add administrative notes or reasons for hiding projects (optional)

Security Considerations

For MVP implementation, the admin page can be secured using:

  • Secret URL (non-discoverable link)
  • Simple password protection
  • Note: Production implementation should use proper authentication

5.5 About Page (about.html) - Optional

Purpose

Provide documentation and information about the ITI Projects Gallery system.

Content Requirements

  • Explanation of the gallery's purpose and objectives
  • Instructions for submitting projects
  • Instructions for editing existing projects
  • Contact information or support channels

6. Database Schema

6.1 Google Sheet Structure

Each row in the Google Sheet represents a single project. The following columns must be implemented:

Column Name Data Type Description Required
Project ID String/Number Unique identifier for each project Yes
Project Title String Name of the project Yes
Team Members String GitHub handles of team members No
Description Text Detailed project description No
Demo / GitHub / App Link URL Links to project resources No
Frameworks / Tech Stack String Technologies used in the project No
Status Enum Not Started, In Progress, or Completed Yes
Email Email Contact email for project editing No
Screenshot URL URL Link to project screenshot No
Visible? Boolean Controls public visibility (TRUE/FALSE) Yes
Admin Notes Text Internal administrative comments No
Created At Timestamp Date and time of project creation Yes
Updated At Timestamp Date and time of last update Yes

7. Project Status Values

The Status field must accept exactly three predefined values:

Status Value Description
Not Started Project has not yet begun development
In Progress Project is currently under active development
Completed Project development has been finished

8. Visibility Control Mechanism

Project visibility in the public gallery is controlled by the Visible field in the Google Sheet:

  • TRUE: Project is visible in the public gallery
  • FALSE: Project is hidden from the public gallery

The admin manually controls this field through the admin interface or directly in the Google Sheet.

9. Edit Authorization Logic

The system implements the following rules for project editing:

  • Users can edit only their own projects
  • Project ownership is verified using Email and Project ID
  • Google Apps Script performs the validation and authorization
  • Updates modify the existing row rather than creating a new entry
  • The gallery automatically reflects changes after Sheet updates

10. System Advantages

  • Eliminates the need for traditional backend infrastructure or database servers
  • Minimal maintenance requirements
  • Students have complete control over their project information
  • Simple admin moderation without complex role-based access control
  • Straightforward migration path to traditional backend systems if needed
  • Zero hosting costs for backend infrastructure
  • Real-time updates without deployment procedures

11. Future Enhancement Opportunities

The following features may be considered for future iterations:

  • Featured projects highlighting system
  • User engagement features (likes, ratings, comments)
  • Project categorization and taxonomy
  • Authentication using ITI institutional email addresses
  • Migration to full backend architecture (Node.js, Django, or similar)
  • Advanced search and filtering capabilities
  • Project analytics and statistics
  • Notification system for project updates

12. Implementation Roadmap

Phase 1: Backend Setup

  1. Create Google Sheet with specified column structure
  2. Develop Google Apps Script API endpoints (doGet and doPost methods)
  3. Implement data validation and error handling in Apps Script
  4. Test API endpoints with sample data

Phase 2: Frontend Development

  1. Develop Gallery Page (index.html) with project display functionality
  2. Implement Project Submission Page (submit.html) with form validation
  3. Create Project Edit Page (edit.html) with data pre-population
  4. Develop optional Admin Page (admin.html) if required
  5. Create optional About Page (about.html) if required

Phase 3: Integration

  1. Connect frontend pages to Google Apps Script API using fetch
  2. Implement error handling and user feedback mechanisms
  3. Add loading states and UI feedback
  4. Test all data flows (create, read, update)

Phase 4: Deployment and Testing

  1. Deploy static frontend to hosting platform
  2. Perform end-to-end testing of all features
  3. Test submission, edit, and visibility control workflows
  4. Conduct user acceptance testing with sample users
  5. Document any issues and implement fixes

Phase 5: Launch

  1. Prepare user documentation and guidelines
  2. Launch system to ITI students
  3. Monitor system performance and user feedback
  4. Implement iterative improvements based on feedback

13. Technical Notes

13.1 Google Apps Script API

The Apps Script must implement the following endpoints:

  • doGet: Retrieve project data (with visibility filtering)
  • doPost: Handle project submissions and updates

13.2 CORS Configuration

Ensure Google Apps Script is configured to accept requests from the deployed frontend domain.

13.3 Data Validation

Implement validation on both client-side (JavaScript) and server-side (Apps Script) to ensure data integrity.

13.4 Error Handling

All API calls must include proper error handling with user-friendly error messages.
